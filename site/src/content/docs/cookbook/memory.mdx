<Aside type="tip" title="Why you need this">
A single `run` is stateless — it forgets everything the moment it returns. Real assistants
need continuity: remember who the user is, what was decided three turns ago, the context of
an ongoing task. Memory gives you multi-turn conversations keyed by id, with a pluggable
store so you can persist transcripts to Redis, Postgres, or anywhere your app already keeps state.
</Aside>

Keep a conversation across turns. JavaScript / Python / Go use a `conversation`
object whose `send` retains history; Java / C# / Elixir use `ask` with a stable id
whose transcript lives in the client's conversation store. Both remember turn 1 at
turn 2.

<Tabs syncKey="lang">
<TabItem label="JavaScript">
```js
const convo = agent.conversation({ toolkit: tk })
await convo.send("My name is Muthu. Remember it.")
const r = await convo.send("What's my name?")
console.log(r.text)   // recalls "Muthu"
```
</TabItem>
<TabItem label="Python">
```python
convo = agent.conversation(tk)
await convo.send("My name is Muthu. Remember it.")
r = await convo.send("What's my name?")
print(r.text)   # recalls "Muthu"
```
</TabItem>
<TabItem label="Go">
```go
convo := agent.Conversation(tk)
convo.Send(ctx, "My name is Muthu. Remember it.")
r, _ := convo.Send(ctx, "What's my name?")
fmt.Println(r.Text)   // recalls "Muthu"
```
</TabItem>
<TabItem label="Java">
```java
agent.ask("My name is Muthu. Remember it.", tk, "user-42");
LlmClient.RunResult r = agent.ask("What's my name?", tk, "user-42");
System.out.println(r.text);   // recalls "Muthu"
```
</TabItem>

The transcript store is pluggable (in-memory by default; bring your own to persist
to Redis, Postgres, …). See [Memory & conversations](/toolnexus/memory/).
