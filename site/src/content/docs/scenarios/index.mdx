The rest of the docs are organised by **primitive** вЂ” MCP, skills, suspension, sub-agents. This

| **"An agent over my real systems, with a human on the risky bits"** вЂ” orders/CRM over MCP, policy as skills, refunds gated by approval, served to a web app | [Support / ops agent](/toolnexus/scenarios/support-agent/) | `mcpConfig` + per-server allowlist В· skills В· `pending`/`waitFor` В· `serve` В· budgets | В§2, В§3, В§10, В§7B |

Nothing here is mutually exclusive. A support agent that remembers a customer is the support
scenario plus `fromDir`; a coding assistant that asks before force-pushing is the coding scenario
plus `waitFor`. The primitives compose because they are all the same two nouns underneath вЂ” a
**`Tool`** and a **loop**.

## Primitives cheat sheet

One line each. Follow the SPEC column when you need the exact contract.

| Primitive | What it gives you | SPEC |
|---|---|---|
| `createToolkit` | The aggregator: MCP servers + skills + your functions + HTTP + built-ins + A2A agents behind one `Tool` interface. | В§4 |
| `createClient` / `run` / `ask` | The host loop вЂ” tool calling, retries, streaming, conversation memory вЂ” against any OpenAI- or Anthropic-style endpoint. | В§8 |
| `agent()` | An agent **is** a tool: system prompt Г— a filtered toolkit view Г— the loop. `.run()` for one-shot, `.asTool()` to drop it into any toolkit. | В§7D |
| `team` + `task` | Listing agents in `team` *is* the sub-agent wiring вЂ” they become the only targets of the child-facing `task` tool. No team в‡’ no delegation (opt-in, non-recursive by default). | В§7D |
| `Budget` | `maxTurns`/`maxTokens`/`maxToolCalls`/`maxWallMs`/`maxChildren`/`maxConcurrent`/`maxDepth`, carved at spawn and re-walked live before every turn. Any stop is loud: `status:"incomplete"` with the limit named. | В§7D |
| `pending` / `waitFor` | A tool returns `Pending(request)` instead of an answer; the host resolves it. Inline with a `waitFor`, or durably as `status:"pending"` you persist and resume later. | В§10 |
| `fromDir` | The directory *is* the agent: `AGENTS.md`, `SOUL.md`, `IDENTITY.md`, `USER.md`, `TOOLS.md`, `HEARTBEAT.md`, `MEMORY.md` compose the system prompt at session start. | В§7E |
| `memory` | An opt-in, file-backed tool with three actions (`add`/`replace`/`remove`) over `MEMORY.md` or `USER.md`. Writes hit disk now; they load into the prompt at the **next** session (frozen-snapshot rule). | В§7E |
| `startAgent` heartbeat | Gives a persona its own clock: each `everyMs` it posts a tick to its own inbox and wakes it to read `HEARTBEAT.md`. A `HEARTBEAT_OK` reply is silent. | В§7E |
| `compactor` | A `beforeLLM` hook that summarizes the older transcript and keeps a recent tail вЂ” tool-pair safe, system prompt preserved. Below `maxTokens` it is a byte-identical no-op. | В§7F |
| `serve` | The inbound direction: expose the toolkit as an **A2A agent** (Agent Card + JSON-RPC, skills-based) and/or as an **MCP server** (`POST /mcp`, tools-based). No auth in core. | В§7B, В§7C |
| Hooks | `beforeLLM` / `afterLLM` / `beforeTool` / `afterTool` вЂ” guardrails, redaction, caching, cost tracking. `beforeTool` can deny or rewrite a call before it runs. | В§8 |

## How to read a scenario page

Each one follows the same spine, so you can skim to the part you need:

1. **The scenario** and a *when to reach for this* table вЂ” including when **not** to.
2. **The wiring at a glance** вЂ” layer в†’ primitive в†’ SPEC section.
3. **Step-by-step build**, with the full example in JavaScript, Python, Go, Java, C# and Elixir.
4. **Honest limits** вЂ” what the library deliberately leaves to you (auth, UI, storage, policy).

<Aside type="note" title="Six ports, one behavior">
Every scenario is expressible in all six languages because parity is the product: the same
`mcp.json` and the same `SKILL.md` folder produce the same tools and the same byte-exact `skill`
output in JavaScript, Python, Go, Java, C# and Elixir (`SPEC.md В§0`). Pick the language your
service is already written in.
</Aside>
