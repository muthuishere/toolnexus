An agent that can only *talk* isn't much use. toolnexus ships opencode's default toolset — **10
built-in tools** — so a fresh toolkit can already *act*: run a command, read and edit files, search,
fetch a URL, ask you a question. `createToolkit()` with no config at all still gives you a working,
acting agent.

## The ten tools

| Tool | Does |
|------|------|
| `bash` | Run a shell command |
| `read` | Read a file |
| `write` | Write a file |
| `edit` | Edit a file in place |
| `grep` | Search file contents |
| `glob` | Find files by pattern |
| `webfetch` | Fetch a URL |
| `question` | [Suspend and ask the user](/toolnexus/suspension/) structured questions |
| `apply_patch` | Apply a patch to the tree |
| `todowrite` | Maintain a task list |

Names and input schemas match opencode. They surface in the tool schema
(`toOpenAI`/`toAnthropic`/`toGemini`) like MCP tools — **not** injected into the system prompt — so
the model discovers them the same way it discovers every other source.

## On by default — with two levels of off

Because `bash`/`write`/`edit`/`apply_patch` run commands and mutate the filesystem, you'll want to
lock them down on untrusted hosts. There's a **whole-source toggle** and a **per-tool map**.

**Turn the whole source off:**

<Tabs syncKey="lang">
<TabItem label="JavaScript">
```ts
const tk = await createToolkit({ builtins: false })
```
</TabItem>
<TabItem label="Python">
```python
tk = await create_toolkit(builtins=False)
```
</TabItem>
<TabItem label="Go">
```go
tk, _ := toolnexus.CreateToolkit(ctx, toolnexus.Options{ Builtins: false })
```
</TabItem>
<TabItem label="Java">
```java
Toolkit tk = Toolkit.create(new Toolkit.Options().builtins(false));
```
</TabItem>
<TabItem label="C#">
```csharp
await using var tk = await Toolkit.CreateAsync(new Toolkit.Options { Builtins = false });
```
</TabItem>
</Tabs>

**Or drop individual tools on the all-on baseline** — here, disable `bash` while keeping the rest:

<Tabs syncKey="lang">
<TabItem label="JavaScript">
```ts
const tk = await createToolkit({ builtins: { tools: { bash: false } } })
```
</TabItem>
<TabItem label="Python">
```python
tk = await create_toolkit(builtins={"tools": {"bash": False}})
```
</TabItem>
<TabItem label="Go">
```go
tk, _ := toolnexus.CreateToolkit(ctx, toolnexus.Options{
    Builtins: toolnexus.BuiltinsConfig{Tools: map[string]bool{"bash": false}},
})
```
</TabItem>
<TabItem label="Java">
```java
Toolkit tk = Toolkit.create(new Toolkit.Options()
        .builtins(Map.of("tools", Map.of("bash", false))));
```
</TabItem>
<TabItem label="C#">
```csharp
await using var tk = await Toolkit.CreateAsync(new Toolkit.Options
{
    Builtins = new Dictionary<string, object?>
    {
        ["tools"] = new Dictionary<string, object?> { ["bash"] = false },
    },
});
```
</TabItem>
</Tabs>

<Aside type="caution" title="Precedence">
A whole-source `false` always wins. On the per-tool map, unknown names are ignored, and other tools
stay on. If one of your own tools has the same name as a builtin, yours shadows it (the builtin is
dropped) — so you can override a built-in tool by registering your own.
</Aside>

The same toggle can come from a top-level `builtins` block in the config the toolkit is built from,
so a deployment can lock the toolset down without touching code.
