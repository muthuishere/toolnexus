## TL;DR

Pointing every framework at the **same mock LLM** and the **same stdio MCP server**, measuring
only the framework's own per-request cost for one fixed tool-calling scenario:

- **toolnexus is the lowest-overhead option in every language measured** — ~1 ms/request in
  Python (7–30× lighter than LangGraph and ADK), ~3 ms/request in Java (~1.8× lighter than
  Spring AI), sub-millisecond in Go — with smaller memory and a dramatically smaller install.

## Methodology

### Why a mock LLM (and not a real one)

The trick to a fair, zero-cost, deterministic comparison: point every framework at the **same
local mock LLM** and the **same stdio MCP server**, and measure only the framework's own cost.
Real model latency (100 ms–seconds) would swamp the thing we actually want to measure — the
framework's tax on top of the model. The mock answers one fixed question with exactly one
tool-calling turn (two parallel tool calls), then a final message, so every run is **2 LLM
round-trips + 2 tool executions** through the framework's own agent loop.

### Metrics and how each was measured

| Metric | How | Tool |
|---|---|---|
| **Cold init** | Wall time to build the toolkit/agent *including* MCP connect + tool discovery. | `time.perf_counter` (Py), `time.Now` (Go), `System.nanoTime` (Java). |
| **Per-request p50/p95/mean** | Wall time of one scenario run, over **N = 30** measured runs after **5 warmup** runs. | same honest timers, in-process. |
| **Peak RSS** | Max resident set of the runner process. | `/usr/bin/time -l` wrapping each runner. |
| **Install size / deps** | Isolated per-framework footprint. | `du -sh` of each venv/jar/binary; declared direct deps. |

## Reproduce it

Full instructions: [`benchmarks/README.md`](https://github.com/muthuishere/toolnexus/tree/main/benchmarks).
In short: install each framework in its own `uv` venv (toolnexus from the local repo), build the
compiled Go/Java runners, start `mock_llm.py`, then `python3 benchmarks/run_all.py` — which runs
each framework under `/usr/bin/time -l`, aggregates mean/p50/p95, and writes `results.json`.

## Caveats — read before quoting

### Go stdio-MCP: fixed in 0.9.2

In the 0.9.0 run the toolnexus **Go** port could not hold a live **stdio** MCP session: the
connect path started the transport on a **timeout context** that `defer cancel()`ed on return,
and `mark3labs/mcp-go` ties the child process to that context — so the MCP child was `SIGKILL`ed
right after `initialize`, and the following `tools/list` hit a dead pipe (isolated, so it failed
*silently*). **Fixed in 0.9.2** by spawning the stdio child under a background context owned by
`client.Close()`; the [resilience benchmark](/toolnexus/resilience/) confirms the Go port now
holds a live stdio session. The Go latency row above still shows the native-tools loop from the
0.9.0 run and will be refreshed with an MCP figure on a re-run.

### Mock-vs-real, and single-machine variance

- These are **framework-overhead** numbers. In production a real model call dwarfs all of them —
  a 5 ms vs 1 ms difference is invisible next to a 500 ms model call per request, but compounds
  under high concurrency, in tight agent loops, and on memory/cost at scale.
- Single machine, single process at a time, otherwise idle. Absolute numbers differ on your
  hardware; the **ratios** between frameworks are the portable takeaway.
- We measured the ports that match each competitor: **Python** (vs LangGraph & ADK) and **Java**
  (vs Spring AI), plus **Go** for reference.

## Verdict

On pure **framework overhead** for a fixed MCP tool-calling scenario, **toolnexus lands at the
low-overhead end in every language measured** — ~1 ms/request in Python (7–30× lighter than
LangGraph and ADK) and ~3 ms/request in Java (~1.8× lighter than Spring AI), sub-millisecond in
Go, with consistently smaller memory and a much smaller install. That matches its design goal: a
small, right-sized library, not a runtime. The honest asterisk — JVM warmup means these figures
describe warm, long-lived agents, which is exactly where framework overhead compounds and matters.
